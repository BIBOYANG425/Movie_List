## 1. Reference semantics

### 1.1 Achievements

- **Tables**: `user_achievements (user_id, badge_key, unlocked_at, PK(user_id,badge_key))` (`supabase_phase4_engagement.sql:138-144`). RLS: SELECT `USING (true)` ‚Äî public incl. anon (`:147-148`); INSERT `WITH CHECK (true)` ‚Äî **any authenticated user can grant any badge to any user** (`:149-150`, B2); no UPDATE/DELETE policies.
- **Catalog**: 16 badges in `AchievementsView.tsx:8-32` (7 milestone, 4 social, 4 taste, 1 special). 15 have grant rules; `early_adopter` (`:31`) has none ‚Äî permanently locked (D1).
- **Grant rules** (`checkAndGrantBadges`, `achievementService.ts:14-125`): threshold counts fetched in one 7-way parallel head-count (`:22-30`). `rankCount` sums movies+tv+books (`:32`) while every badge string says "movies" (D2); genre badges read `user_rankings.genres` only (`:63-73`, movie-only); tier badges (`s_tier_10`/`d_tier_5`) movie-only (`:76-85`); `first_list` counts `movie_lists.created_by` (`:27`); **`first_review`/`review_10` count the legacy `movie_reviews` table** (`:24`) ‚Äî reviews have lived in `journal_entries.review_text` since C2 (`reviewService.ts:83` "Query from journal_entries instead of movie_reviews"), so review badges are effectively ungrantable for post-C2 users (D3).
- **Granting is client-side and lazy**: fires only when the OWN Achievements tab mounts (`AchievementsView.tsx:54-57`). Insert result is unchecked, no `onConflict` handling (`:88-90`); milestone feed events then fire unconditionally per badge (`:93-118` ‚Üí `logMilestoneEvent`, `feedService.ts:722-741`, `event_type='milestone'`). Race/abuse surface: two devices ‚Üí PK 23505 ‚Üí whole multi-badge insert fails silently while BOTH devices emit milestone events ‚Üí duplicate feed cards with (transiently) no badge row (B3). Badges self-heal on next visit; the events don't.
- **Surfaces**: ProfilePage `achievements` tab (`ProfilePage.tsx:603-607`) ‚Äî **passes `user.id` (the viewer), not `profile.id`** (B1). Feed: `FeedMilestoneCard` with the 3/day global throttle (`feedService.ts:319-324`). Notifications: `badge_unlock` is in the type CHECK and styled in the bell (`NotificationBell.tsx:11`) but **nothing ever writes it** ‚Äî the only notification writers in the codebase are `followService.ts:14` (`new_follower`) and `journalService.ts:279` (`journal_tag`); `createNotification` (`notificationService.ts:5`) has zero callers (D4).
- **iOS**: nothing. No achievements repository/screen; `NotificationRepository.swift:39` decodes `badge_unlock` (render-ready, write-dead); FeedCards already maps `milestone` (`FeedCards.swift:31,112`) with the same daily cap (`FeedPipeline.swift:79-81`).

### 1.3 Letterboxd import (the reference pipeline)

Flow: ZIP ‚Üí JSZip (`letterboxdImportService.ts:8,88-120`) ‚Üí `parseCSV` (RFC 4180, quoted fields/embedded newlines, `csvParser.ts:5-83`) ‚Üí merge/dedup ‚Üí preview ‚Üí TMDB resolution ‚Üí tier mapping ‚Üí position assignment ‚Üí batch persist.

- **ZIP extraction**: five known files (`ratings/watched/diary/reviews/watchlist.csv`, `:92-98`); rows mapped to `LetterboxdRawEntry` (`Name/Year/Letterboxd URI/Rating/Watched Date|Date/Review/Rewatch`, `:107-116`).
- **Merge** (`:124-184`): key = `name.lower()|year`; source priority reviews(4) > diary(3) > ratings(2) > watched(1); lower-priority sources fill gaps (`:156-161`); watchlist excludes anything rated/watched (`:164-180`).
- **Tier map** (`mapRatingToTier :69-76`): ‚â•4.5‚ÜíS, ‚â•3.5‚ÜíA, ‚â•2.5‚ÜíB, ‚â•1.5‚ÜíC, else D; null rating ‚Üí not ranked.
- **Resolution** (`:222-374`): proxy-routed `search/movie` (api key server-side, `proxyRequest :244`), year-exact preferred (`:255-260`), genres via `GENRE_MAP` ‚â§3 (`:262-265`); **typo-variant retry loop on zero results** (`typoRetryVariants`, `:308-313` ‚Äî the PR #41 integration); non-2xx = `'http-error'` ‚Üí exponential backoff, MAX_RETRIES 3, never sprays variants after an HTTP error (`:296-303`); CONCURRENCY 8, 50 ms inter-batch delay (`:222-224`); abortable (modal Cancel, `LetterboxdImportModal.tsx:330-338`).
- **Positions** (`assignPositions :378-409`): per-tier sort rating desc ‚Üí alphabetical, `rank_position` 0..n, `bracket = classifyBracket(genres)`.
- **Persist** (`persistImport :415-554`): ids are **canonical `tmdb_{n}`** via `canonicalMovieTmdbId` (`:445,455,482-486,521` ‚Äî the C3 convention, write-values pinned by `letterboxdImportIds.test.ts`); positions APPEND after the user's current per-tier max (`:432-441` ‚Äî density preserved, no `set_tier_order` needed); batch-50 upserts with `onConflict: 'user_id,tmdb_id', ignoreDuplicates: true` into `user_rankings` (`:470-472`), `watchlist_items` (`:503-505`, ranked items excluded `:482-487`), `journal_entries` (rows with review text or watched date, `:515-544`). **Import writes NO activity events and NO stubs** ‚Äî a silent bulk import (pin as deliberate; D7).
- **UI** (`LetterboxdImportModal.tsx`): `<input type="file" accept=".zip">` (`:233-239`), 10 MB cap (`:30`), preview step (tier distribution, counts), progress bar with live title + cancel, results with `failedResolutions` list. Entry point: ProfilePage "Import from Letterboxd" (`ProfilePage.tsx:511-517`, own profile only).
- **Known soft spots**: imported/journal counters `+= batch.length` even when `ignoreDuplicates` dropped rows (`:477,549` ‚Äî overcount, D5); journal `watched_date` fallback is UTC `new Date().toISOString().split('T')[0]` (`:531` ‚Äî the C2 ledger item (c), still open, D6); **unrated entries are resolved then dropped entirely** ‚Äî an unrated Letterboxd review is never imported although the preview's `reviewCount` counts it (`buildPreview :207` vs modal filter `:159`, D8) and its TMDB calls are wasted.
- **iOS**: nothing. `Package.swift` has ONE dependency (supabase-swift, `:12`) ‚Äî ZIPFoundation would be a new dependency; the file picker needs SwiftUI `fileImporter`; the pure pieces (`mapRatingToTier`, merge, positions) are mechanical ports; resolution can reuse `TMDBService` + the proxy. ## 2. Findings

### Blocking (web fixes required BEFORE iOS copies the flows)

**B1 ‚Äî ProfilePage's Lists and Achievements tabs render the VIEWER's data on ANY profile.**
`ProfilePage.tsx:599` passes `userId={user.id}` to `MovieListView` and `:605` passes `userId={user.id}` to `AchievementsView` ‚Äî `user` is the signed-in viewer, while every other tab uses `profile.id` (journal `:561`, memories `:585,592`). Viewing a friend's profile (reachable ‚Äî `/profile/:profileId`, gated only by `canSeeFullProfile :89`) shows YOUR lists and YOUR badge grid presented as theirs; `isOwnProfile={profile.isSelf}` merely suppresses the grant check, so the wrong data renders read-only. The reference semantics iOS would copy ("profile tabs show the profile owner's content") are broken on web.
*Fix shape:* `profile.id` for both; `AchievementsView` already takes `isOwnProfile` for the grant gate; `MovieListView` needs a viewer-vs-owner distinction anyway once B4 is resolved.

**B2 ‚Äî Anyone can grant any badge to any user: `user_achievements` INSERT RLS is `WITH CHECK (true)`.**
`supabase_phase4_engagement.sql:149-150`. Combined with client-side granting (¬ß1.1) this is a wide-open abuse surface: a curl with any user's anon session can mint `rank_100` for anyone (SELECT is public, so fake badges are also publicly visible), and the milestone feed event that "announces" it is client-optional. iOS must NOT copy client-side granting; the web reference must move granting server-side first ‚Äî a SECURITY DEFINER RPC (`grant_earned_badges()`) that recomputes thresholds from the caller's own rows and inserts `ON CONFLICT DO NOTHING`, or at minimum `WITH CHECK (user_id = auth.uid())` to stop cross-user grants while keeping self-grant honesty client-side.

**B3 ‚Äî `checkAndGrantBadges` ignores the insert result and double-fires milestone events.**
`achievementService.ts:88-90` (`await supabase.from('user_achievements').insert(rows)` ‚Äî error unchecked, no `onConflict`) then `:93-118` logs one `milestone` activity event per "new" badge unconditionally. Two concurrent mounts (two tabs/devices ‚Äî the check runs on every Achievements-tab visit) both compute the same `newBadges`, one insert fails wholesale on the PK, both emit feed events ‚Üí duplicate milestone cards, and on the loser the badges appear granted (events) but aren't (rows) until the next visit re-heals. Same event-after-unconfirmed-write class as C5-B4.
*Fix shape:* `upsert(..., { onConflict: 'user_id,badge_key', ignoreDuplicates: true })` + emit events only for rows actually inserted (or fold into the B2 RPC which returns the granted keys ‚Äî one fix for both).

**B4 ‚Äî Curated lists are a shell: no UI can ever put a movie IN a list, and "my lists" shows everyone's public lists.**
`addMovieListItem` (`socialService.ts:363-378`) has zero callers ‚Äî `MovieListView.tsx:8` imports it and never invokes it; there is no "add to list" affordance in UniversalSearch, MediaDetailModal, TierRow, or anywhere else. `deleteMovieList` is equally unreachable (`handleDelete` defined `:74-81`, wired to nothing). Meanwhile `getMyMovieLists` (`:311-341`) merges ALL public lists from every user into the owner's "Movie Lists" tab with `itemCount` never computed (permanent "0 movies", `MovieListView.tsx:260`). Every list in production is therefore empty, and the `list_create` feed card advertises `itemCount: 0`. Porting this to iOS as-is would duplicate a non-feature.
*Fix shape (product decision first, Q4):* add an "add to list" action (MediaDetailModal and/or a picker from the tier row), wire delete, split "my lists" from "discover lists" (or drop the public merge), compute `itemCount` via a count aggregate ‚Äî OR descope lists from C7 entirely and delete the tab.

### Deferred

**D1 ‚Äî `early_adopter` is in the catalog but no grant rule exists** (`AchievementsView.tsx:31`) ‚Äî renders permanently locked; grant at signup during beta or drop it.
**D2 ‚Äî Badge copy vs counting basis mismatch:** `rankCount` includes tv+books (`achievementService.ts:32`) but every rank badge says "movies"; genre/tier badges are movie-only (`:64,77`). Pin the cross-media rule in the contract before iOS re-implements the thresholds.
**D3 ‚Äî Review badges count the legacy `movie_reviews` table** (`:24`); reviews live in `journal_entries` since C2 (`reviewService.ts:83`) ‚Äî `first_review`/`review_10` are ungrantable for post-C2 users. Count `journal_entries WHERE review_text IS NOT NULL` instead.
**D4 ‚Äî `badge_unlock` and `list_like` notification types are dead on the write side** (only `new_follower` + `journal_tag` are ever written; `createNotification` has no callers). Either write them (badge grant ‚Üí recipient bell; list like ‚Üí creator bell) or drop them from the CHECK + bell styling. iOS bell already renders both kinds if they ever arrive.
**D5 ‚Äî Import result counters overcount:** `rankingsImported`/`journalImported` add the full batch even when `ignoreDuplicates` skipped rows (`letterboxdImportService.ts:477,549`); the results screen can claim more imports than happened.
**D6 ‚Äî Import journal dates fall back to UTC "today"** (`:531`) ‚Äî the C2 ledger item (c); one-line `localDateString` fix, do it in C7's web pass.
**D7 ‚Äî Import emits no events/stubs** ‚Äî deliberate anti-spam posture but unpinned; add to `docs/contracts/shared-payloads.md` so the iOS port doesn't "helpfully" emit 300 `ranking_add`s.
**D8 ‚Äî Unrated Letterboxd reviews are silently dropped:** preview counts them (`:207`), persist filters them out (`LetterboxdImportModal.tsx:159`), and their TMDB resolutions are wasted. Either journal unrated reviews (no ranking) or say so in the preview.
**D9 ‚Äî `/u/:username` OG meta is client-rendered** (`PublicProfilePage.tsx:64-91`) ‚Äî crawlers get the SPA shell (vercel rewrite), so shared links never unfurl. Needs SSR/edge-function OG or prerender; same hosting surface as the AASA work.
**D10 ‚Äî `spool://` scheme hijack TODO** (`Info.plist:66-70`) ‚Äî universal links are the mitigation; if UL infra stalls (Q2), at least move OAuth to a reverse-DNS scheme.
